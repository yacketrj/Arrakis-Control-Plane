## Industry-standard release alignment

The current release process is aligned with common industry release-management expectations in these areas:

- semantic-versioning-style version labels
- immutable Git tags for release points
- release-candidate flow before final release
- release checklist with validation, rollback, approval, and risk sections
- compact changelog plus durable per-slice records
- explicit known-risk acceptance for RC scope
- upstream attribution preservation

Current gaps before a stronger final/stable release:

- release artifacts are not yet signed
- artifact checksums are not yet generated and published
- SBOM generation is deferred
- SAST, DAST, secret scanning, and vulnerability scan evidence is deferred
- GitHub Release artifact attachment is not automated
- post-release verification evidence is still manual
- Bash and PowerShell update-script modularization is not fully complete

## Rollback plan

```text
Rollback trigger:
- release candidate validation fails in a way that cannot be fixed safely before tagging
- new critical/high security issue is identified
- update script cannot produce repeatable build artifacts

Rollback steps:
1. Do not create or publish the tag if tagging has not occurred.
2. If tagged, delete local and remote v0.1.0-rc.1 tag only if it has not been consumed externally.
3. Revert release setup commits if necessary.
4. Return VERSION to the prior value or dev state if release is abandoned.
5. Record the failed candidate in PATCH_NOTES.md and the release checklist.

Expected rollback duration:
15-30 minutes for docs/config-only rollback.

Data compatibility concerns:
None for release setup. No schema or player-data mutation was added.

Operator communications:
Notify operators that v0.1.0-rc.1 is deferred or superseded and continue using the previous validated commit.
```
